# Update migrations

One-off steps that an update needs **beyond swapping code** — most importantly,
regenerating per-install infrastructure the self-updater never touches (the updater
does `git reset + npm ci + pm2 restart`; it does **not** re-run `install.sh`, so e.g.
`private.Caddyfile` is never regenerated by an update).

## How it works

`app/api/update/route.js` runs pending migrations after `npm ci`, before the pm2
restart. Each migration runs **once per install** and is recorded by filename in
`<data>/migrations-applied.json` (the data dir survives `git reset`).

- **Filename:** `NNN-slug.sh` or `NNN-slug.js` — zero-padded ordinal, applied in
  lexicographic order. Anything not matching `^\d{3,}[-_].+\.(sh|js)$` is ignored
  (so this README and `template.sh` never run).
- **Execution:** `bash <file>` (or `node <file>` for `.js`), `cwd` = install root,
  with `.env.local` present and `PATH` including the bundled Node. Exit `0` = success
  (recorded); non-zero **halts** the migration phase and is surfaced in the update job
  (`job.migrationError`) — and is retried on the next update.

## Hard rules

1. **Best-effort, never blocking.** The runner guarantees the code update + restart
   happen even if a migration fails. Do not rely on a migration to make the app boot.
2. **Idempotent.** Assume it may run more than once (e.g. if recording failed). Check
   before acting; make re-runs a no-op.
3. **Own your safety.** For service configs (Caddy, cloudflared, pm2): write to a temp
   file, **validate**, back up the current file, then atomically swap and reload. On
   any failure, restore the backup and exit non-zero — never leave a broken config
   live (the app itself is proxied through Caddy, so a bad swap can lock users out).
4. **Reuse existing state, don't re-derive.** Read ports/usernames/animals from the
   running config or `.env.local`; never re-run a free-port scan or re-pick values.

Copy `template.sh` to start a new migration.
